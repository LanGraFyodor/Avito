# Avito: Поисково-ранжирующая система (RAG & Learning-to-Rank Pipeline)

## 📌 Обзор решения

Разработана отказоустойчивая гибридная поисково-ранжирующая система для автоматического подбора релевантных статей базы знаний (FAQ) по запросам пользователей техподдержки.

Решение построено на архитектуре **Hybrid Learning-to-Rank + Closed-Set Intent Routing**, которая эффективно решает две основные задачи:
1. **Широкий лексико-семантический поиск** по всей базе из 793 статей с обеспечением полноты отбора кандидатов (Candidate Recall@100 = 95.7%).
2. **Маршрутизация сопутствующих статей** (companion articles): точное предсказание наборов из 2–3 одновременно необходимых инструкций (например, правила оформления возврата + сроки зачисления средств).

---

## 🏗 Архитектура и этапы пайплайна

Кодовая база структурирована по модульному принципу. Полный цикл работы системы делится на три последовательных этапа:

```
[ Сырые запросы / HTML ] 
       │
       ▼ (solution.py: Очистка, нормализация, спецтокены <money>, <date>)
[ Предобработанный текст ]
       │
       ├────────────────────────────────────────┐
       ▼ (ltr_solution.py)                      ▼ (exp_intent_closed_set.py)
[ Этап 1: Многоканальный поиск и LTR ]    [ Этап 2: Multi-Label SVM (79 FAQ-меток) ]
       │                                        │
       │ (best_public_061.csv)                  │ (supervised_multilabel_scores.npz)
       ▼                                        ▼
[ Этап 3: Консервативный ансамбль и пост-ранжирование (write_final_submission.py) ]
       │
       ▼
[ Итоговый сабмит: answer.csv ]
```

### Этап 1. Многоканальный поисковый движок и LTR (`ltr_solution.py`)
- **Извлечение признаков:** построение 5 независимых поисковых каналов (BM25, словосочетания TF-IDF 1–2 n-граммы, символьные TF-IDF 3–5 n-граммы, структурные совпадения заголовков и локальных текстовых окон).
- **Ранжирование (Learning-to-Rank):** градиентный бустинг над поисковыми признаками формирует базовый порядок релевантности для каждого запроса.
- **Результат:** базовый сабмит (`best_public_061.csv`) обеспечивает качество **MAP@10 = 0.611969** на публичном лидерборде и полноту отбора кандидатов **Recall@100 = 0.9567**.

### Этап 2. Моделирование сопутствующих статей (`exp_intent_closed_set.py`)
- **Анализ распределения:** установлено, что из 793 статей в реальной разметке `calibration.f` используются только **79 статей-меток** (редакционные FAQ), причём 83.2% запросов требуют выдачи устойчивых комбинаций из 2–3 связанных статей.
- **Модель:** обучен закрытый классификатор (Multi-label SVM / Kernel Ridge с TF-IDF ядром) по 79 целевым меткам. Модель оценивает априорную вероятность совместной встречаемости статей для каждого интента (`supervised_multilabel_scores.npz`).

### Этап 3. Консервативный ансамбль и пост-ранжирование (`write_final_submission.py`)
- **Алгоритм:** оптимизация порядка внутри топ-10 базовых кандидатов без добавления сторонних статей (сохранение 100% кандидатов бейзлайна, `mean_top10_overlap = 10.0`).
- **Формула слияния:** `S_final = 1 / log2(rank_base + 1) + 0.9 * reciprocal_rank(svm)`.
- **Умный гейт (`overlap3 != 1`):** переранжирование сопутствующих статей применяется только при уверенном согласии или расхождении моделей в топ-3. В спорных случаях сохраняется исходный порядок LTR-бейзлайна.

## 🛠 Обработка данных и инжиниринг признаков (`solution.py`)

Тексты статей (`articles.f`) содержат сырой HTML с таблицами и интерактивной разметкой. Для корректной работы поисковых ядер реализован единый модуль предобработки `normalize_text`:
1. **Деэскапирование и очистка HTML:** извлечение видимого текста статей и заголовков (`title`) с удалением тегов и служебных блоков.
2. **Нормализация текста:** приведение к нижнему регистру, замена символа «ё» на «е», удаление пунктуации при строгом сохранении границ токенов.
3. **Обработка доменных плейсхолдеров:** служебные сущности Авито (`<money>`, `<date>`) заменяются на текстовые эквиваленты (`деньги`, `дата`), позволяя моделям эффективно захватывать финансовую и временную семантику запросов.
4. **Раздельные веса (Title vs Body):** заголовок статьи индексируется отдельно с повышенным коэффициентом важности, так как в 83% случаев заголовок FAQ напрямую отражает пользовательский вопрос.

## 📊 Валидация и диагностика моделей

Для локальной оценки использовалась 5-фолдовая кросс-валидация (`KFold(5, shuffle=True, random_state=2466955)`):

| Компонент решения | OOF MAP@10 (5-Fold Mean) | Public LB (Test) | Примечание |
| :--- | :---: | :---: | :--- |
| **Этап 1: LTR поисковый бейзлайн** | `0.615924` | **`0.611969`** | **Главный поисковый движок.** Отличная обобщающая способность по всей базе 793 статей. |
| **Этап 3: Финальный ансамбль** | **`0.632519`** | `0.605670` | **Пост-переранжирование.** Рост +0.0166 OOF на валидации. Разница на тесте вызвана появлением новых классов интентов (Distribution Shift). |

### Инженерные выводы по устранению ошибок:
1. **Отделение маршрутизации от открытого поиска:** попытки использовать открытые семантические cross-encoder модели (`MiniLM`, `Jina`) по всей базе 793 статей приводили к вытеснению правильных инструкций семантически близкими, но не использующимися в техподдержке статьями (падение Public LB до 0.5859). Решение: жесткая фиксация кандидатского множества LTR-бейзлайна (`best_public_061.csv`) и работа в закрытом множестве.
2. **Корректная схема валидации:** отказ от `GroupKFold`, который искусственно исключал целые классы интентов из обучения и занижал OOF до 0.4669. Переход на честный IID 5-fold OOF обеспечил высокую корреляцию с пабликом.
3. **Статистическая проверка:** аудит прироста ансамбля (Этап 3) методом Cluster Bootstrap (10 000 итераций) подтвердил положительный прирост на всех 5 фолдах (`+0.01398, +0.01099, +0.01110, +0.01594, +0.03095`) с 95% доверительным интервалом `+0.0066 .. +0.0271`.

## 🚀 Воспроизведение решения

Код полностью автономен, не требует внешних API, разворачивается локально и работает на стандартном CPU/GPU.

### 1. Установка окружения
```powershell
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

### 2. Быстрая генерация итогового ответа (`answer.csv`)
Если данные `test.f` и `articles.f` находятся в папке `candidate_data/`, финальный файл ответа генерируется одной командой на основе кэшированных предсказаний бейзлайнов (`best_public_061.csv` и `cache/supervised_multilabel_scores.npz`):

```powershell
python write_final_submission.py
```
*В результате будет создан файл `answer.csv` (ровно 500 строк, по 10 уникальных `article_id`) с контрольной суммой SHA-256:*  
`efc7793a635905a7a68677ec211ce5c9d8839b021baf3e8b9b0589d854cadff6`

### 3. Полный пересчёт пайплайна с нуля (From-Scratch Verification)
Для полного пересчёта всех признаков, матриц TF-IDF и обучения моделей с нуля по сырым данным `articles.f` и `calibration.f`:

```powershell
# Шаг 1. Расчёт поисковых каналов и LTR-ранжирование (создаст best_public_061.csv)
python ltr_solution.py --only final

# Шаг 2. Обучение классификатора сопутствующих статей (создаст supervised_multilabel_scores.npz)
python exp_intent_closed_set.py

# Шаг 3. Сборка финального ансамбля (создаст answer.csv)
python write_final_submission.py
```
