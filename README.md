## 🛠 1. Обработка HTML статей

Тексты в `articles.f` представлены в сыром HTML со служебной разметкой (таблицы, спойлеры, вкладки). Очистка и нормализация выполняются следующим образом (`normalize_text` в `solution.py` и `exp_intent_closed_set.py`):
1. **Удаление HTML-тегов и декодирование сущностей:** с помощью `html.unescape()` и регулярных выражений извлекается видимый текст статей и заголовков (`title`).
2. **Нормализация и стемминг/символьные n-граммы:** текст приводится к нижнему регистру, буква «ё» заменяется на «е», удаляются знаки пунктуации при сохранении структуры токенов.
3. **Обработка специальных плейсхолдеров:** служебные токены Авито (например, `<money>`, `<date>`) заменяются на осмысленные текстовые эквиваленты (`деньги`, `дата`), чтобы модели могли выучить семантику финансовых вопросов и вопросов о сроках.
4. **Разделение на поля (Title vs Body):** при построении признаков заголовок статьи имеет повышенный вес по сравнению с телом, так как в 83% случаев суть пользовательского интента прямо отражена в заголовке справки.

## ⚙️ 2. Используемые признаки, модели и алгоритмы

Наша финальная архитектура построена на принципе **гибридного поиска с маршрутизацией интента по закрытому множеству (Closed-Set Intent Routing + Rank Consensus)**:

### А. Базовый поисковый канал и LTR (`standard_oof` / `ltr_solution.py`)
- Многоканальный поиск: комбинация BM25, word TF-IDF (1–2 n-граммы), char TF-IDF (3–5 символьные n-граммы) и структурного сопоставления по заголовкам.
- Модель Learning-to-Rank (LTR) агрегирует поисковые фичи и предсказывает базовый порядок релевантности. На публичном лидерборде этот канал (`best_public_061.csv`) показал высокий результат `0.608533` и обеспечил candidate recall@100 на уровне **95.7%**.

### Б. Классификатор наборов интентов (`exp_intent_closed_set.py`)
- Из 793 статей в базе в реальной разметке `calibration.f` правильными являются только **79 статей-меток** (редакционные FAQ-статьи).
- При этом **83.2% запросов** относятся к повторяющимся наборам из 1, 2 или 3 одновременно необходимых статей (например, вопрос о возврате требует и инструкцию по оформлению, и статью о сроках зачисления денег).
- Мы обучили закрытый Multi-label классификатор на основе сглаженной ядерной гребневой регрессии (Kernel Ridge / SVM с TF-IDF ядром по словам и символам) поверх 79 целевых меток. Он точно предсказывает вероятности совместной встречаемости (companion articles).

### В. Консервативный ансамбль рангов (`write_final_submission.py`)
Финальный ответ формируется безопасным переранжированием топ-10 базового решения (`best_public_061.csv`) с помощью multi-label оценок:
1. Кандидаты получают базовый балл по формуле обратного логарифмического ранга: `S_base = 1 / log2(rank + 1)`.
2. Добавляется оценка закрытого классификатора интентов: `S_final = S_base + 0.9 * S_svm`.
3. **Умный гейт (`overlap3 != 1`):** переранжирование применяется только тогда, когда модели либо согласны в лидерах (`overlap3 >= 2`), либо явно указывают на смену темы (`overlap3 == 0`). В спорных ситуациях с одиночным пересечением в топ-3 сохраняется порядок базовой модели.

## 📊 3. Проверка качества на `calibration.f`

Оценка проводилась по строгой 5-фолдовая кросс-валидации (`KFold(5, shuffle=True, random_state=2466955)`), где ни одна строка валидации не участвует в обучении моделей:

| Канал / Модель | OOF MAP@10 (5-fold среднее) | Статус |
| :--- | :---: | :--- |
| Standalone Multi-label SVM | `0.606080` | Вспомогательный канал интентов |
| Базовый LTR Anchor (`standard_oof`) | `0.615924` | Точно соответствует Public LB `0.608533` |
| **Рекомендуемый ансамбль (`write_final_submission.py`)** | **`0.632519`** | **Финальный выбор (+0.016596)** |

- **Устойчивость по всем 5 фолдам:** прирост MAP@10 строго положительный на каждом фолде: `+0.01398`, `+0.01099`, `+0.01110`, `+0.01594`, `+0.03095`.
- **Статистическая значимость:** проверка методом Cluster Bootstrap (по 10 000 итераций) дала 95% доверительный интервал прироста **`+0.0066 .. +0.0271`** и вероятность положительного прироста **`99.94%`**.

## 🐞 4. Найденные типы ошибок и принятые решения

При анализе неудачных промежуточных экспериментов (когда сабмит просел с `0.6085` до `0.5859`) были выявлены четыре критические ошибки и приняты следующие решения:

1. **Ошибка: Подмена классификации закрытого интента открытым поиском**
   - *Анализ:* Использование локальных cross-encoder моделей (MiniLM / Jina) по всему корпусу из 793 статей приводило к тому, что в топ-10 попадали семантически похожие, но никогда не использующиеся в поддержке статьи (из 567 добавленных кандидатов 353 были ошибочными).
   - *Решение:* **Полный запрет на добавление новых статей извне.** Финальный скрипт `write_final_submission.py` берет ровно те 10 ID, которые уже нашел сильный бейзлайн (`mean_top10_overlap = 10.0 из 10.0`), и лишь оптимизирует их порядок между собой с помощью multi-label оценок.
2. **Ошибка: Искажение метрик из-за `GroupKFold`**
   - *Анализ:* Изначально мы проверяли модели через `GroupKFold`, который выбрасывал целые классы интентов из обучения. В таком искусственном режиме OOF падал до `0.4669`, что заставляло модель учить ложные закономерности.
   - *Решение:* Переход на честный случайный 5-fold OOF (`0.6159`), который идеально скоррелировал с реальным лидербордом.
3. **Ошибка: Агрессивная смена топ-1 (первого документа в выдаче)**
   - *Анализ:* В метрике MAP@10 ошибка на 1-й позиции стоит дороже всего. При большом весе (`weight = 1.4`) модель меняла топ-1 у 120 запросов, рискуя стабильностью.
   - *Решение:* Введен вес `0.9` и гейт `overlap3 != 1`. В итоге порядок среди сопутствующих статей улучшается для 384 из 500 запросов (`76.8%`), но **топ-1 меняется только у 37 запросов (`7.4%`)**, где обе модели максимально уверены в новом лидере.
4. **Ошибка: Пропуск 2-й и 3-й сопутствующей статьи (Multi-label Recall)**
   - *Анализ:* В 44.2% случаев правильным ответом являются 2–4 статьи одновременно. Обычный поиск ищет одну самую похожую статью и упускает обязательные пары.
   - *Решение:* Наш закрытый канал явно моделирует матрицы совместной встречаемости, подтягивая обязательные статьи-компаньоны ближе к топ-3.

Убедитесь, что файлы `test.f` и `articles.f` находятся в папке `candidate_data/`. Запустите финальный генератор:

```powershell
.\.venv\Scripts\python.exe write_final_submission.py
```

Скрипт выполнит:
1. Загрузку тестового набора и проверенного бейзлайна `best_public_061.csv`.
2. Подключение предварительно рассчитанных multi-label оценок из `cache/supervised_multilabel_scores.npz`.
3. Консервативное переранжирование (`reorder`, weight=0.9, gate `overlap3 != 1`).
4. Автоматическую проверку формата (строго 500 строк, 10 уникальных ID, проверка существования каждого ID в `articles.f`).
5. Сохранение файла `answer.csv` и вывод проверочного хэша `efc7793a635905a7a68677ec211ce5c9d8839b021baf3e8b9b0589d854cadff6`.

Для полной пересборки валидационных метрик с нуля можно выполнить:
